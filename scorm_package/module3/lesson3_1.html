This lesson establishes a comprehensive Financial Operations (FinOps) framework that aligns engineering decisions with business outcomes, ensuring your AWS environment delivers maximum value from day one.

1. Establishing a FinOps Practice

Financial Operations (FinOps) is the cultural practice of bringing financial accountability to cloud spending. Organizations that implement FinOps practices typically achieve 20-40% cost savings within the first 90 days of adoption. The FinOps lifecycle consists of three iterative phases: Inform (visibility and allocation), Optimize (rates and usage), and Operate (continuous governance).

A successful FinOps practice requires cross-functional collaboration between engineering, finance, and procurement teams. Start by establishing a Cloud Cost Center of Excellence (CCoE) that owns tagging standards, budget thresholds, and anomaly detection rules. Use AWS Cost Explorer, AWS Budgets, and Cost Anomaly Detection to create a single pane of glass for financial visibility across your entire AWS footprint.

  • Inform phase: Implement a comprehensive tagging strategy with mandatory tags for Environment, Team, Project, and CostCenter. Enforce via AWS Organizations Service Control Policies (SCPs).
  • Optimize phase: Systematically right-size resources, purchase commitment-based discounts, and eliminate waste.
  • Operate phase: Automate governance with AWS Budgets alerts, Lambda-based remediation, and monthly FinOps review cadences.

2. Right-Sizing Compute Resources

Compute costs typically represent 60-70% of total cloud spend, making right-sizing the single most impactful optimization lever. After migration from GCP, workloads are frequently over-provisioned because teams map GCP machine types to the nearest AWS equivalent without performance benchmarking.

AWS Compute Optimizer

AWS Compute Optimizer uses machine learning to analyze 14 days of historical CloudWatch metrics and deliver instance-level recommendations. Organizations that act on Compute Optimizer findings achieve an average cost reduction of 25% on their EC2 spend. The service evaluates CPU utilization, memory pressure, network I/O, and disk throughput to recommend optimal instance families and sizes.

Establish a recurring review cadence using the following utilization thresholds:

  • <40% average CPU utilization → Downsize to the next smaller instance type
  • 40-80% average CPU utilization → Current sizing is appropriate; continue monitoring
  • >80% average CPU utilization → Upsize or implement horizontal auto-scaling
# Example: Query Compute Optimizer for EC2 recommendations
aws compute-optimizer get-ec2-instance-recommendations \
    --filters "name=Finding,values=OVER_PROVISIONED" \
    --output table \
    --query 'instanceRecommendations[*].{
        InstanceId: instanceArn,
        Current: currentInstanceType,
        Recommended: recommendationOptions[0].instanceType,
        Savings: recommendationOptions[0].estimatedMonthlySavings.value
    }'

3. Commitment-Based Discounts

AWS offers two primary commitment vehicles: Savings Plans and Reserved Instances (RIs). Both deliver discounts of up to 72% compared to On-Demand pricing in exchange for a one- or three-year usage commitment.

Savings Plans vs. Reserved Instances

Compute Savings Plans offer the greatest flexibility, automatically applying discounts across instance families, sizes, operating systems, tenancies, and AWS regions. Unlike Reserved Instances, which are locked to a specific instance type in a specific Availability Zone, Compute Savings Plans adapt as your architecture evolves. This flexibility makes them the recommended starting point for most organizations, especially those still stabilizing their AWS footprint post-migration.

Optimal Coverage Strategy

The recommended coverage ratio balances savings with operational agility:

  • 70-80% baseline coverage with Savings Plans: Cover your predictable, steady-state compute demand. Use Cost Explorer's Savings Plans recommendations to identify the optimal hourly commitment.
  • 20-30% On-Demand and Spot Instances: Reserve this capacity for burst workloads, development environments, and fault-tolerant batch processing.
Tip: Commitment Purchase Timing

Do not purchase Savings Plans or Reserved Instances immediately after migration. Allow 30-60 days of stabilization to establish accurate utilization baselines in your new AWS environment. Premature commitments based on GCP usage patterns frequently result in stranded reservations when workloads are right-sized. Use Cost Explorer's Savings Plans recommendations engine only after your utilization patterns have normalized.

4. Spot Instances for Fault-Tolerant Workloads

AWS Spot Instances provide access to spare EC2 capacity at discounts of up to 90% compared to On-Demand pricing. The average interruption rate across all instance types and regions is approximately 5%, making Spot viable for a broad range of workloads when combined with proper fault-tolerance patterns.

Ideal Spot candidates include CI/CD pipelines, data processing jobs, batch rendering, machine learning training, and stateless web application tiers behind Auto Scaling Groups. Use EC2 Fleet or Spot Fleet with capacity-optimized allocation strategy to diversify across multiple instance types and Availability Zones, minimizing interruption risk. 

# Example: Launch a diversified Spot Fleet
aws ec2 request-spot-fleet --spot-fleet-request-config '{
    "AllocationStrategy": "capacityOptimized",
    "TargetCapacity": 10,
    "IamFleetRole": "arn:aws:iam::role/spot-fleet-role",
    "LaunchSpecifications": [
        {"InstanceType": "m5.xlarge", "SubnetId": "subnet-abc123"},
        {"InstanceType": "m5a.xlarge", "SubnetId": "subnet-abc123"},
        {"InstanceType": "m5d.xlarge", "SubnetId": "subnet-def456"},
        {"InstanceType": "m6i.xlarge", "SubnetId": "subnet-def456"}
    ]
}'

5. Storage Cost Optimization

S3 Intelligent-Tiering

Amazon S3 Intelligent-Tiering automatically moves objects between access tiers based on observed access patterns, delivering 40-60% storage cost savings with zero retrieval fees and no operational overhead. This is the recommended default storage class for data with unpredictable or changing access patterns migrated from Google Cloud Storage.

Lifecycle Policies

For data with well-understood access patterns, implement explicit lifecycle rules to transition objects through progressively cheaper storage classes:

  • Day 0-30: S3 Standard — frequent access, lowest latency
  • Day 30: Transition to S3 Standard-IA (Infrequent Access) — ~45% cheaper, per-retrieval fee
  • Day 90: Transition to S3 Glacier Instant Retrieval — ~68% cheaper than Standard, millisecond retrieval
  • Day 180: Transition to S3 Glacier Deep Archive — ~95% cheaper than Standard, 12-hour retrieval

EBS Optimization

Elastic Block Store (EBS) volumes are a common source of waste. Conduct a monthly audit to identify and delete unattached volumes — these accumulate when EC2 instances are terminated but their volumes are retained. Additionally, convert legacy gp2 volumes to gp3, which delivers the same baseline performance at a 20% lower price point while offering independently configurable IOPS and throughput. 

# Find unattached EBS volumes
aws ec2 describe-volumes \
    --filters "Name=status,Values=available" \
    --query 'Volumes[*].{ID:VolumeId,Size:Size,Created:CreateTime}' \
    --output table

Key Takeaways

  • Implement a FinOps practice with cross-functional ownership to achieve 20-40% cost savings within 90 days of migration.
  • Right-size compute resources using AWS Compute Optimizer; compute represents 60-70% of total spend and yields an average 25% reduction.
  • Cover 70-80% of baseline compute with Savings Plans for flexibility across instance families, sizes, and regions, but wait 30-60 days post-migration before purchasing commitments.
  • Deploy Spot Instances (up to 90% savings) for fault-tolerant workloads using diversified capacity-optimized fleets.
  • Enable S3 Intelligent-Tiering for 40-60% storage savings, implement lifecycle policies for known access patterns, and audit EBS volumes monthly for waste.
  • Cost optimization is a continuous discipline — automate governance with AWS Budgets, anomaly detection, and scheduled right-sizing reviews.
