Module 3 — Lesson 2

Performance Tuning and Architectural Optimization

8 min read

A successful migration from GCP to AWS is not complete until workloads are optimized for the target platform's unique performance characteristics. Simply lifting and shifting workloads preserves functional parity but leaves significant performance gains on the table. This lesson applies the AWS Well-Architected Framework's Performance Efficiency pillar to systematically tune networking, database, and caching layers, transforming a migrated environment into a truly cloud-native architecture.

The Performance Efficiency pillar is built on four key areas: selection of the right resource types, monitoring to ensure sustained performance, understanding of trade-offs, and the ability to adopt new services as they become available. Post-migration optimization should address each of these areas across every tier of the application stack.

1. Network Performance Optimization

Enhanced Networking (ENA)

AWS Enhanced Networking uses single root I/O virtualization (SR-IOV) to provide high-performance networking capabilities on supported instance types. Enabling the Elastic Network Adapter (ENA) delivers bandwidth of up to 100 Gbps on current-generation instance families including M5, C5, R5, and newer (M6i, C6i, R6i). This represents a substantial improvement over the standard Xen-based virtual networking stack and is essential for data-intensive workloads migrated from GCP, where similar capabilities were provided via gVNIC.

Verify ENA support and enablement on all migrated instances: 

# Verify ENA support on an instance
aws ec2 describe-instances --instance-ids i-0abcd1234efgh5678 \
    --query 'Reservations[*].Instances[*].EnaSupport'

# Verify ENA driver version inside the instance
modinfo ena | grep version

VPC Endpoints for AWS Services

Traffic between your VPC and AWS services like S3 and DynamoDB typically traverses the public internet via a NAT Gateway, introducing latency and data transfer costs. VPC Gateway Endpoints (for S3 and DynamoDB) and VPC Interface Endpoints (for other services) route traffic over the AWS private backbone, achieving a 30-50% reduction in latency while eliminating NAT Gateway processing charges. 

# Create a VPC Gateway Endpoint for S3
aws ec2 create-vpc-endpoint \
    --vpc-id vpc-0abc123def456 \
    --service-name com.amazonaws.us-east-1.s3 \
    --route-table-ids rtb-0abc123def456

# Create an Interface Endpoint for DynamoDB
aws ec2 create-vpc-endpoint \
    --vpc-id vpc-0abc123def456 \
    --vpc-endpoint-type Interface \
    --service-name com.amazonaws.us-east-1.dynamodb \
    --subnet-ids subnet-aaa111 subnet-bbb222

Multi-AZ Deployment with Application Load Balancer

Distribute workloads across multiple Availability Zones using an Application Load Balancer (ALB) with cross-zone load balancing enabled. This configuration ensures traffic is evenly distributed across targets in all registered AZs, preventing hotspots that degrade performance when one zone has fewer healthy instances. Cross-zone load balancing also improves fault tolerance by transparently routing around unhealthy targets regardless of their zone.

  • Deploy targets in a minimum of three Availability Zones for production workloads
  • Enable cross-zone load balancing on the ALB target group to equalize request distribution
  • Use connection draining (deregistration delay) of 30-60 seconds to allow in-flight requests to complete during scaling events
  • Configure health check intervals of 10-15 seconds with a threshold of 2 consecutive failures for rapid unhealthy target removal

2. Database Performance Tuning

RDS Performance Insights

Amazon RDS Performance Insights provides a dashboard for monitoring database load and identifying performance bottlenecks. It visualizes active sessions sliced by wait events, SQL statements, hosts, and users, making it straightforward to pinpoint whether performance issues stem from lock contention, inefficient queries, or resource saturation. Enable Performance Insights on all RDS instances migrated from Cloud SQL to gain immediate visibility into database behavior under production load.

Read Replicas for Read-Heavy Workloads

Applications migrated from Cloud SQL often have read-heavy access patterns that overload a single primary instance. Deploy RDS Read Replicas to offload 40-60% of read traffic from the primary database. Read replicas use asynchronous replication and can be placed in different Availability Zones or even different regions for disaster recovery. Route read queries using application-level logic, a database proxy, or connection string configuration.

RDS Proxy for Connection Management

Serverless and containerized architectures frequently overwhelm relational databases with excessive connection overhead. Amazon RDS Proxy pools and shares database connections, reducing connection overhead by up to 80% and enabling 10x connection scaling compared to direct connections. RDS Proxy also improves failover times by automatically routing connections to a standby instance without dropping the application's connection pool. 

# Create an RDS Proxy for a PostgreSQL database
aws rds create-db-proxy \
    --db-proxy-name migration-app-proxy \
    --engine-family POSTGRESQL \
    --auth '{
        "AuthScheme": "SECRETS",
        "SecretArn": "arn:aws:secretsmanager:us-east-1:123456789:secret:db-creds",
        "IAMAuth": "REQUIRED"
    }' \
    --role-arn arn:aws:iam::123456789:role/rds-proxy-role \
    --vpc-subnet-ids subnet-aaa111 subnet-bbb222 subnet-ccc333

Aurora PostgreSQL: The Cloud SQL Migration Target

For workloads migrating from Google Cloud SQL for PostgreSQL, Amazon Aurora PostgreSQL is the recommended target database engine. Aurora's distributed, log-structured storage architecture delivers 3x the throughput of standard PostgreSQL while maintaining full wire-protocol compatibility. Aurora supports up to 15 auto-scaled read replicas with sub-10ms replica lag, compared to Cloud SQL's maximum of 10 read replicas with higher replication latency.

Aurora's Distributed Architecture Advantage

Aurora's storage layer automatically replicates data six ways across three Availability Zones, with continuous backup to Amazon S3. This architecture eliminates the need for manual multi-AZ replication configuration that Cloud SQL requires. Aurora can sustain the loss of an entire Availability Zone with zero data loss and automatically heals storage faults in the background. The distributed storage also enables instant crash recovery — Aurora replays redo logs in parallel on demand rather than requiring sequential log replay, reducing restart times from minutes to seconds.

3. Caching Layer Optimization

Amazon ElastiCache

Introducing an in-memory caching layer between your application and database tiers can dramatically reduce latency and database load. Amazon ElastiCache supports both Redis and Memcached engines, delivering microsecond read latency and reducing direct database queries by 70-90% for cached data paths.

  • ElastiCache for Redis: Choose Redis for advanced data structures, persistence, pub/sub messaging, Lua scripting, and cluster mode with automatic sharding. Ideal for session stores, leaderboards, real-time analytics, and rate limiting.
  • ElastiCache for Memcached: Choose Memcached for simple key-value caching with multi-threaded performance. Best for database query result caching and HTML fragment caching where persistence is not required.

Implement a cache-aside (lazy loading) pattern for most use cases: the application checks the cache first, retrieves from the database on a cache miss, and writes the result to the cache with a TTL. This pattern balances data freshness with performance. 

# Cache-aside pattern (Python pseudocode)
import redis

cache = redis.Redis(host='migration-cache.abc123.ng.0001.use1.cache.amazonaws.com')

def get_product(product_id):
    cache_key = f"product:{product_id}"

    # Check cache first
    cached = cache.get(cache_key)
    if cached:
        return json.loads(cached)

    # Populate cache with 5-minute TTL
    cache.setex(cache_key, 300, json.dumps(product))
    return product

Amazon CloudFront CDN

Amazon CloudFront is a global content delivery network with over 410 edge locations across 90+ cities worldwide. Placing CloudFront in front of your application tier delivers a 50-80% reduction in end-user latency and offloads 60-70% of requests from origin servers. CloudFront supports dynamic content acceleration (using persistent connections and TCP optimizations to the origin), static asset caching, WebSocket connections, and Lambda@Edge for edge-side compute.

API Gateway Caching

For RESTful APIs migrated from GCP's API Gateway or Cloud Endpoints, enable Amazon API Gateway caching to store endpoint responses in an in-memory cache. This delivers a 10-100x performance improvement for repeated requests by eliminating backend invocations entirely. Configure cache TTLs per-method and use cache key parameters to ensure correct cache segmentation. API Gateway caching is available in cache sizes from 0.5 GB to 237 GB.

Key Takeaways

  • Apply the AWS Well-Architected Performance Efficiency pillar systematically across networking, database, and caching layers after migration.
  • Enable Enhanced Networking (ENA) on all production instances for up to 100 Gbps bandwidth, and deploy VPC endpoints to reduce service-to-service latency by 30-50%.
  • Use RDS Proxy to reduce connection overhead by 80% and enable 10x connection scaling for serverless and containerized workloads.
  • Migrate Cloud SQL workloads to Aurora PostgreSQL for 3x throughput and up to 15 auto-scaled read replicas with sub-10ms lag.
  • Deploy ElastiCache for microsecond-latency caching that reduces database queries by 70-90%, and use CloudFront's 410+ edge locations to cut end-user latency by 50-80%.
  • Enable API Gateway caching for 10-100x improvement on frequently requested API endpoints, eliminating redundant backend invocations.
