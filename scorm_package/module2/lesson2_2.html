Data Migration Strategies and Transfer Optimization

8 min read

Data migration is consistently ranked as the highest-risk component of any cloud-to-cloud transition. Unlike infrastructure, which can be reproduced from code, data is unique, irreplaceable, and often subject to regulatory constraints around residency and encryption in transit. A failed data migration can result in data loss, extended downtime, compliance violations, and eroded stakeholder confidence. This lesson provides a comprehensive, strategy-by-strategy breakdown of how to move data from GCP storage and analytics services to their AWS equivalents with maximum throughput, minimum risk, and validated integrity.

1. Object Storage Migration: GCS to Amazon S3

Google Cloud Storage (GCS) and Amazon S3 share a nearly identical object storage paradigm—flat namespace, bucket-level policies, object versioning, and lifecycle rules. This similarity makes object storage one of the more straightforward migration targets, but the sheer volume of data in production environments demands purpose-built tooling rather than manual copy operations.

AWS DataSync is the recommended service for GCS-to-S3 transfers. DataSync is a fully managed data transfer service that automates moving data between on-premises storage, other clouds, and AWS services. For cross-cloud object storage migration, DataSync is up to 10x faster than open-source tools such as rclone or gsutil rsync, thanks to its purpose-built network protocol, automatic parallel transfers, and built-in data integrity verification.

A notable reference case is Thomson Reuters, which migrated approximately 250 TB of data in 72 hours using AWS DataSync. The key enablers for this performance were:

  • Deploying multiple DataSync agents in parallel to saturate available bandwidth
  • Configuring task-level bandwidth throttling to avoid saturating production network links
  • Using DataSync's built-in verification (checksum comparison after transfer) rather than post-hoc validation scripts
  • Scheduling transfers during off-peak hours to minimize impact on production GCS read performance

DataSync Configuration Essentials

# AWS CLI: Create a DataSync task for GCS-to-S3 migration
# Step 1 - Create source location (GCS via S3-compatible API)
aws datasync create-location-object-storage \
  --server-hostname storage.googleapis.com \
  --bucket-name my-gcs-source-bucket \
  --access-key GOOG1EXXXXXXXXXXXXX \
  --secret-key xXxXxXxXxXxXxXxXxXxX \
  --server-protocol HTTPS \
  --server-port 443

# Step 2 - Create destination location (S3)
aws datasync create-location-s3 \
  --s3-bucket-arn arn:aws:s3:::my-aws-target-bucket \
  --s3-config BucketAccessRoleArn=arn:aws:iam::123456789012:role/DataSyncS3Role

# Step 3 - Create and start the task
aws datasync create-task \
  --source-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-src \
  --destination-location-arn arn:aws:datasync:us-east-1:123456789012:location/loc-dst \
  --options VerifyMode=POINT_IN_TIME_CONSISTENT,OverwriteMode=ALWAYS

2. Database Migration with AWS DMS

For relational databases (Cloud SQL for PostgreSQL/MySQL migrating to Amazon RDS or Aurora), AWS Database Migration Service (DMS) provides continuous replication with Change Data Capture (CDC). CDC enables near-zero-downtime migration by replicating ongoing changes from the source database to the target while the initial full load is in progress and thereafter.

DMS Sizing and Performance Guidelines

Parameter Development / Test Production
Instance class dms.c5.large dms.c5.2xlarge or larger
Storage 50 GB SSD 200–500 GB SSD
Target replication lag < 30 seconds < 5 seconds
Max tables per task Unlimited 8–10 (split large schemas into parallel tasks)
LOB mode Full LOB Limited LOB (set max size to reduce overhead)
Multi-AZ Disabled Enabled (automatic failover)

For production workloads, always use c5.2xlarge or larger replication instances. Smaller instances may appear to work during initial testing but will fall behind under CDC load when the source database is under heavy write pressure. The target replication lag for production cutover readiness is less than 5 seconds—this ensures the cutover window (stop writes, let DMS catch up, switch connection strings) can be completed within minutes rather than hours.

3. Analytics Migration: BigQuery to Amazon Redshift

Important: Heterogeneous Migration Complexity

BigQuery-to-Redshift migration is classified as a heterogeneous migration—the most complex category. These platforms differ not only in SQL dialect and data types but also in their fundamental architecture. BigQuery is serverless with columnar storage and automatic scaling; Redshift is a provisioned cluster with distribution keys and sort keys that must be explicitly designed. There is no one-click migration path. Plan for 4–8 weeks of dedicated effort for enterprise-scale data warehouses.

The AWS Schema Conversion Tool (SCT) partially automates the schema translation from BigQuery to Redshift, but significant manual intervention is required for:

  • SQL dialect differences: BigQuery's Standard SQL supports STRUCT, ARRAY, and nested/repeated fields natively. Redshift has no equivalent—these must be flattened into separate tables or serialized as JSON VARCHAR columns.
  • User-Defined Functions (UDFs): BigQuery JavaScript UDFs must be rewritten as Redshift Python or SQL UDFs.
  • Partitioning: BigQuery's ingestion-time and column-based partitioning translates to Redshift's DISTKEY and SORTKEY design, which requires query pattern analysis.
  • Materialized views: Logic must be validated and rewritten to account for Redshift's different refresh semantics.

Staged Migration Approach

The recommended approach for large-scale BigQuery-to-Redshift migration uses a staged pipeline that avoids direct service-to-service transfer (which is not natively supported):

  1. Export from BigQuery to GCS: Use bq extract to export tables in Apache Parquet format. Parquet preserves column types, supports compression, and is splittable for parallel processing.
  2. Transfer GCS to S3 via DataSync: Use the same DataSync pipeline established for object storage migration. Parquet files transfer efficiently due to columnar compression.
  3. Load from S3 into Redshift: Use the Redshift COPY command to bulk-load Parquet files directly from S3.

Redshift COPY Performance: Format Comparison

File Format Load Speed Compression Schema Preservation Recommendation
Apache Parquet ~100 MB/s per node Excellent (Snappy/ZSTD) Full type fidelity Preferred
CSV (gzip) ~25 MB/s per node Good (gzip) Requires manifest Legacy fallback
JSON ~15 MB/s per node Moderate Flexible but lossy Semi-structured only
Avro ~80 MB/s per node Good (Deflate) Good type support Alternative to Parquet

Parquet loading achieves approximately 100 MB/s per Redshift node, compared to only 25 MB/s for CSV—a 4x performance advantage. For a 10 TB data warehouse on an 8-node ra3.xlplus cluster, this translates to roughly 13 minutes for Parquet versus nearly an hour for CSV. 

-- Redshift COPY command for Parquet loading from S3
COPY analytics.fact_transactions
FROM 's3://migration-staging-bucket/bigquery-export/fact_transactions/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftS3ReadRole'
FORMAT AS PARQUET
COMPUPDATE ON
STATUPDATE ON;

5. Data Validation and Integrity Checks

No data migration is complete without comprehensive validation. Every transfer must be verified at multiple levels:

  • Object count validation: Compare the number of objects in GCS buckets against S3 buckets post-transfer
  • Byte-level checksums: DataSync performs automatic MD5/SHA-256 verification; for DMS, run row-count and checksum queries on source and target databases
  • Schema validation: Confirm that all tables, columns, indexes, and constraints exist in the target with correct data types
  • Sampling validation: Run spot-check queries on 1–5% of records to verify data content accuracy
  • Referential integrity: Validate foreign key relationships in the target database match the source

Key Takeaways

  • Data migration is the highest-risk component of cloud migration. Use purpose-built tools (DataSync, DMS, SCT) rather than manual scripts to ensure reliability and performance.
  • AWS DataSync achieves up to 10x faster GCS-to-S3 transfers than open-source alternatives, with built-in integrity verification. The Thomson Reuters case study demonstrates 250 TB transferred in 72 hours.
  • For relational database migration, size DMS replication instances at c5.2xlarge or larger for production workloads, targeting less than 5 seconds of replication lag for cutover readiness.
  • BigQuery-to-Redshift is the most complex migration path. Use a staged approach: BigQuery export to Parquet on GCS, DataSync to S3, then Redshift COPY. Parquet loads at 100 MB/s per node versus 25 MB/s for CSV.
  • Always validate data integrity at multiple levels—object counts, checksums, schema verification, sampling, and referential integrity—before proceeding to application cutover.
