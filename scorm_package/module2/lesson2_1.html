Module 2 — Lesson 1

Infrastructure-as-Code Conversion and Environment Provisioning

8 min read

Migrating cloud infrastructure between providers is far more than a lift-and-shift exercise. The underlying primitives—networking constructs, security policies, resource naming conventions, and provisioning APIs—differ fundamentally between Google Cloud Platform and Amazon Web Services. In this lesson, we examine how Infrastructure-as-Code (IaC) tooling provides the automation backbone for a reliable, repeatable, and auditable migration from GCP to AWS. We will walk through provider conversions, networking architecture translation, and security configuration best practices that form the foundation of your target-state environment.

1. The Terraform Multi-Cloud Advantage

According to the HashiCorp 2023 State of Cloud Strategy Survey, 76% of enterprises now use Terraform as their primary IaC tool for multi-cloud provisioning. This dominance is not accidental. Terraform's provider-based architecture allows engineering teams to express infrastructure intent in a single declarative language (HCL) while targeting any cloud API through swappable provider plugins.

For organizations currently using GCP's Deployment Manager, the migration to AWS presents an additional conversion step. Deployment Manager templates (written in YAML or Jinja2/Python) have no direct equivalent in AWS. While AWS CloudFormation serves a similar purpose, the template syntax, intrinsic functions, and resource type schemas are entirely different. There is no automated converter between Deployment Manager and CloudFormation—manual effort is required for every template. This is a primary reason we recommend consolidating on Terraform before beginning the migration, as it decouples your IaC definitions from any single provider's proprietary tooling.

2. Terraform Provider Block Conversion

The most fundamental change in a GCP-to-AWS Terraform migration is the provider block itself. Every resource definition must be re-mapped from google_* resource types to their aws_* equivalents. Below is a side-by-side comparison that illustrates the scope of this change.

GCP Provider and Compute Instance

# GCP Terraform Configuration
provider "google" {
  project = "my-gcp-project-id"
  region  = "us-central1"
  zone    = "us-central1-a"
}

resource "google_compute_instance" "app_server" {
  name         = "app-server-01"
  machine_type = "e2-standard-4"
  zone         = "us-central1-a"

  boot_disk {
    initialize_params {
      image = "debian-cloud/debian-11"
      size  = 50
    }
  }

  network_interface {
    network    = google_compute_network.main.id
    subnetwork = google_compute_subnetwork.private.id
  }

  tags = ["app-server", "http-server"]
}

AWS Provider and EC2 Instance (Equivalent)

# AWS Terraform Configuration
provider "aws" {
  region  = "us-east-1"
  profile = "migration-admin"
}

resource "aws_instance" "app_server" {
  ami           = "ami-0c55b159cbfafe1f0"  # Debian 11
  instance_type = "m5.xlarge"

  subnet_id              = aws_subnet.private[0].id
  vpc_security_group_ids = [aws_security_group.app_tier.id]

  root_block_device {
    volume_size = 50
    volume_type = "gp3"
  }

  tags = {
    Name = "app-server-01"
    Tier = "application"
  }
}

Key differences to note: GCP uses project scoping while AWS uses region and IAM profile scoping. GCP assigns firewall rules via network tags, whereas AWS uses explicit security group associations. Machine type naming conventions also differ—GCP's e2-standard-4 maps approximately to AWS's m5.xlarge (4 vCPUs, 16 GB RAM).

3. Networking: GCP Global VPC to AWS Regional VPC

Warning: Architectural Paradigm Shift

GCP Virtual Private Clouds are global resources—a single VPC spans all regions, and subnets are regional. AWS VPCs, by contrast, are strictly regional. A single GCP global VPC may need to be decomposed into multiple AWS VPCs (one per region), connected via VPC Peering or AWS Transit Gateway. Failing to account for this difference is one of the most common and costly architectural mistakes in cross-cloud migrations.

The target-state AWS networking architecture should follow AWS Well-Architected Framework recommendations. For a production workload, the minimum viable VPC design includes:

  • CIDR block: A /16 VPC CIDR (e.g., 10.0.0.0/16) providing 65,536 IP addresses
  • Subnets: /24 subnets (256 addresses each), with both public and private subnets in a minimum of 3 Availability Zones for high availability
  • Internet Gateway: Attached to the VPC for public subnet egress/ingress
  • NAT Gateways: One per AZ in public subnets, providing outbound internet access for private subnets without exposing them to inbound traffic
  • Route tables: Separate route tables for public subnets (default route to IGW) and private subnets (default route to NAT Gateway)

Terraform AWS VPC Example

# AWS VPC with Multi-AZ Subnet Layout
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = { Name = "migration-vpc-prod" }
}

resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id
  tags   = { Name = "migration-igw" }
}

# Public subnets across 3 AZs
resource "aws_subnet" "public" {
  count                   = 3
  vpc_id                  = aws_vpc.main.id
  cidr_block              = cidrsubnet("10.0.0.0/16", 8, count.index)
  availability_zone       = data.aws_availability_zones.az.names[count.index]
  map_public_ip_on_launch = true

  tags = { Name = "public-${data.aws_availability_zones.az.names[count.index]}" }
}

# Private subnets across 3 AZs
resource "aws_subnet" "private" {
  count             = 3
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet("10.0.0.0/16", 8, count.index + 10)
  availability_zone = data.aws_availability_zones.az.names[count.index]

  tags = { Name = "private-${data.aws_availability_zones.az.names[count.index]}" }
}

# NAT Gateway (one per AZ for HA)
resource "aws_eip" "nat" {
  count  = 3
  domain = "vpc"
}

resource "aws_nat_gateway" "nat" {
  count         = 3
  allocation_id = aws_eip.nat[count.index].id
  subnet_id     = aws_subnet.public[count.index].id

  tags = { Name = "nat-${data.aws_availability_zones.az.names[count.index]}" }
}

4. Security Groups vs. GCP Firewall Rules

GCP firewall rules are applied at the VPC network level and target instances via network tags or service accounts. AWS security groups are stateful, VPC-specific firewalls attached directly to elastic network interfaces (ENIs). The stateful nature means that return traffic for an allowed inbound rule is automatically permitted—no explicit outbound rule is needed for response packets.

A well-designed AWS security group hierarchy for a migrated application typically follows a three-tier model:

Security Group Inbound Rules Source Purpose
Bastion SG TCP 443 (SSM) VPC Endpoints Administrative access via Session Manager
Application Tier SG TCP 8080, 443 ALB Security Group Receives traffic from load balancer only
Database Tier SG TCP 5432 (PostgreSQL) Application Tier SG Accepts connections only from app servers

This layered approach ensures that each tier can only communicate with its adjacent tier, implementing the principle of least privilege at the network level.

5. Replacing SSH with AWS Systems Manager Session Manager

A critical security improvement during migration is eliminating direct SSH access to instances. AWS Systems Manager Session Manager provides encrypted, audited shell access without opening inbound SSH ports (TCP 22). According to AWS security benchmarks, this approach reduces the attack surface by approximately 40% compared to traditional bastion-host-with-SSH architectures.

Session Manager advantages over SSH-based access:

  • No inbound ports required—instances initiate outbound connections to SSM endpoints
  • Full session logging to CloudWatch Logs and S3 for compliance auditing
  • IAM-based access control—no SSH key distribution or rotation overhead
  • Centralized access management through IAM policies and AWS Organizations SCPs
  • Session recording for forensic analysis and change management evidence

To enable Session Manager, instances require the SSM Agent (pre-installed on Amazon Linux 2 and recent Ubuntu AMIs), an instance profile with the AmazonSSMManagedInstanceCore managed policy, and VPC endpoints for ssm, ssmmessages, and ec2messages services (or NAT Gateway egress).

Key Takeaways

  • Terraform's provider-based architecture makes it the ideal IaC tool for cross-cloud migration, with 76% enterprise adoption (HashiCorp 2023). Consolidate on Terraform before migrating to avoid manual Deployment Manager-to-CloudFormation conversion.
  • GCP's global VPC model must be decomposed into AWS regional VPCs—this is a fundamental architectural change that affects subnet design, routing, and inter-region connectivity.
  • Design AWS VPCs with a /16 CIDR block, /24 subnets across a minimum of 3 Availability Zones, with Internet Gateways for public subnets and NAT Gateways for private subnet egress.
  • AWS security groups are stateful and VPC-specific. Implement a three-tier hierarchy (bastion, application, database) following the principle of least privilege.
  • Replace SSH with AWS Systems Manager Session Manager to reduce attack surface by approximately 40%, eliminate inbound port requirements, and gain centralized IAM-based access control with full audit logging.
