Application Cutover and Validation Procedures

8 min read

The cutover phase is the culmination of months of planning, infrastructure provisioning, and data migration. It is also the moment of highest organizational risk. Research from Gartner and the Cloud Industry Forum consistently shows that migrations using structured cutover methodologies achieve a 68% success rate, compared to only 34% for ad-hoc approaches—a 2x improvement that underscores the value of rigorous planning. This lesson provides a step-by-step framework for DNS-based traffic shifting, database cutover procedures, validation checklists, and post-cutover hypercare operations.

1. DNS-Based Traffic Shifting with Route 53

Rather than performing a single "big bang" cutover, best practice dictates a gradual traffic shift using DNS-level controls. Amazon Route 53 weighted routing policies allow you to split traffic between the existing GCP endpoints and the new AWS endpoints by assigning percentage weights to each record set. This approach provides a built-in rollback mechanism: if the AWS environment exhibits problems, you simply shift the weight back to GCP.

The traffic shifting follows a blue-green deployment pattern. The GCP environment (blue) continues serving production traffic while the AWS environment (green) receives an increasing share. At any point during the ramp-up, the blue environment can resume 100% of traffic within the TTL propagation window (typically 60–300 seconds).

Traffic Shifting Phases

Phase AWS Weight GCP Weight Duration Validation Gate
Phase 0: Baseline 0% 100% — All pre-cutover checks pass
Phase 1: Canary 10% 90% 2–4 hours Error rate < 0.5%, latency within baseline
Phase 2: Early Adoption 25% 75% 4–8 hours No P1/P2 incidents, database replication healthy
Phase 3: Majority 50% 50% 12–24 hours Performance parity confirmed, all integrations verified
Phase 4: Near-Complete 75% 25% 24–48 hours Cost metrics within forecast, no data discrepancies
Phase 5: Full Cutover 100% 0% Permanent All validation categories green for 24+ hours

2. Database Cutover via Read Replica Promotion

Database cutover is the most latency-sensitive operation during migration. The recommended strategy uses a read replica promotion approach that minimizes the write-unavailability window to minutes rather than hours:

  1. Verify replication lag < 5 seconds on the AWS DMS task. Monitor the CDCLatencyTarget CloudWatch metric continuously for the 30 minutes preceding cutover.
  2. Enable application read-only mode by toggling a feature flag or deploying a maintenance page. This stops all write operations to the GCP source database.
  3. Wait for DMS replication to fully catch up (lag = 0 seconds). This typically takes 10–60 seconds after writes stop, depending on the final transaction batch size.
  4. Stop the DMS task and run a final row-count and checksum validation between source and target databases. Use table-level COUNT(*) and CHECKSUM aggregations.
  5. Promote the AWS RDS instance from replica to standalone primary, update application connection strings (via AWS Secrets Manager rotation or environment variable update), and disable read-only mode. The application is now writing to AWS.

The total write-unavailability window using this approach is typically 2–5 minutes for databases under 500 GB. For larger databases, consider splitting into per-schema DMS tasks to parallelize the final catch-up verification.

3. Validation Procedures

Post-cutover validation must be systematic, not ad-hoc. We define five validation categories, each with specific pass/fail criteria:

Validation Category Matrix

Category Method Tools Pass Criteria
Functionality End-to-end test suite execution Selenium WebDriver, Cypress 100% of critical path tests pass
Performance Load test at 120% peak traffic Apache JMeter, Gatling p95 latency within 10% of GCP baseline
Data Reconciliation Row counts, checksums, sampling Custom SQL scripts, AWS Glue 100% row count match, 99.99% checksum match
Security Vulnerability scan, IAM audit AWS Inspector, Prowler No critical/high findings, IAM least-privilege verified
Integration Third-party API and service connectivity Postman collections, synthetic monitors All integration endpoints responding within SLA

Automated testing is non-negotiable for cutover validation. Selenium WebDriver scripts should cover every critical user journey (login, core transactions, data retrieval, payment flows). Apache JMeter load tests should simulate at least 120% of observed peak traffic to establish performance headroom. These tests must be pre-built and validated against the GCP environment before cutover day, so that any test failures on AWS can be attributed to the migration rather than test defects.

4. Post-Cutover Hypercare Phase

Tip: Hypercare Best Practices

The hypercare phase should last 7 to 14 days following full cutover. During this period, maintain the GCP environment in a warm standby state—do not decommission any GCP resources until hypercare concludes successfully. Staff a dedicated war room (physical or virtual) with representatives from infrastructure, application development, database administration, security, and business operations for at least the first 72 hours. After the initial 72-hour intensive period, transition to an on-call rotation with 15-minute response SLAs for the remainder of the hypercare window.

CloudWatch Monitoring and Alarm Configuration

During hypercare, every production metric must be monitored against dynamically calculated baselines. The recommended approach for alarm thresholds is mean + 2 standard deviations, calculated from the first 48 hours of AWS production traffic. This statistical approach avoids both false positives (thresholds too tight) and missed incidents (thresholds too loose). 

# CloudWatch alarm: API latency anomaly detection
aws cloudwatch put-metric-alarm \
  --alarm-name "migration-api-latency-anomaly" \
  --metric-name "TargetResponseTime" \
  --namespace "AWS/ApplicationELB" \
  --statistic "Average" \
  --period 300 \
  --evaluation-periods 3 \
  --threshold 0.850 \
  --comparison-operator "GreaterThanThreshold" \
  --dimensions Name=LoadBalancer,Value=app/migration-alb/1234567890 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:migration-alerts \
  --treat-missing-data "breaching"

Rollback Decision Triggers

Rollback decisions must be based on objective, measurable criteria—not subjective assessment. Define these triggers before cutover and ensure all stakeholders agree:

  • Error rate trigger: A 5% increase in error rate (5xx responses) compared to the GCP baseline, sustained over a 15-minute evaluation window
  • Latency trigger: A 20% degradation in p95 latency compared to the GCP baseline, sustained over a 15-minute evaluation window
  • Data integrity trigger: Any confirmed data loss or corruption event, regardless of scope
  • Security trigger: Any active exploitation or critical vulnerability discovered in the AWS environment
  • Business trigger: Revenue-impacting defects confirmed by business stakeholders

If any trigger fires, the rollback procedure is: shift Route 53 weights back to 100% GCP within 60 seconds, stop all DMS replication tasks, and convene the war room to assess root cause before re-attempting.

5. War Room Procedures

The war room is the command center during cutover and hypercare. Effective war room operations require:

  • Escalation paths: Define a three-tier escalation matrix—L1 (on-call engineer, 5-minute response), L2 (team lead, 15-minute response), L3 (VP Engineering / CTO, 30-minute response). Every tier must have a primary and backup contact with confirmed availability during the cutover window.
  • Communication templates: Pre-draft templates for stakeholder updates at each phase transition: "Cutover Phase X initiated," "Validation gate passed/failed," "Rollback initiated," and "Cutover complete." Templates reduce cognitive load during high-pressure situations.
  • Decision log: Maintain a real-time, timestamped log of every decision, who made it, and the rationale. This log is essential for post-mortem analysis and compliance documentation.
  • Runbook execution: Every action during cutover must reference a specific step in the pre-approved runbook. Deviations from the runbook require L2 or L3 approval.
  • Communication cadence: Issue stakeholder status updates every 30 minutes during active cutover, every 2 hours during the first 24 hours of hypercare, and daily thereafter.

Key Takeaways

  • Structured cutover methodologies achieve a 68% success rate versus 34% for ad-hoc approaches. Invest in detailed runbooks, validation gates, and pre-defined rollback criteria before cutover day.
  • Use Route 53 weighted routing to gradually shift traffic from GCP to AWS in phases (10% → 25% → 50% → 75% → 100%), with explicit validation gates between each phase. This blue-green pattern enables instant rollback.
  • Database cutover via read replica promotion minimizes write-unavailability to 2–5 minutes. Ensure DMS replication lag is under 5 seconds before initiating the cutover sequence.
  • Validate across five categories post-cutover: functionality (Selenium), performance (JMeter at 120% peak), data reconciliation (row counts and checksums), security (Inspector/Prowler), and integration (synthetic monitors).
  • Maintain a 7–14 day hypercare phase with CloudWatch alarms set at mean + 2 standard deviations. Define objective rollback triggers: 5% error rate increase or 20% latency degradation sustained over 15 minutes. Keep GCP in warm standby until hypercare concludes.
