Module 1 — Lesson 3

Application Portfolio Analysis and Wave Planning

8 min read

Before a single workload moves from GCP to AWS, the migration team must answer two foundational questions: What exactly are we moving? and In what order? Application Portfolio Analysis provides the answer to the first question by cataloging every workload, its dependencies, and its optimal migration strategy. Wave Planning answers the second by organizing workloads into sequenced migration batches that minimize risk, maximize learning, and maintain business continuity. Together, these disciplines transform a chaotic "lift everything at once" impulse into a disciplined, repeatable execution framework.

1. The 6R Migration Strategy Framework

Originally introduced by Gartner and popularized by AWS, the 6R framework classifies every application into one of six migration strategies. Each strategy represents a different trade-off between migration speed, cost, and post-migration optimization potential. The classification decision for each application should be made collaboratively between the migration team, application owners, and architecture reviewers.

Strategy Description Typical Share Effort Level When to Use
Rehost (Lift & Shift) Move the application as-is to AWS with minimal changes. GCE VMs become EC2 instances; storage volumes migrate to EBS. 30–40% Low Legacy applications, time-sensitive migrations, applications with no active development team
Replatform (Lift & Optimize) Make targeted optimizations during migration without changing core architecture. Example: Cloud SQL → Aurora, GKE manifests → EKS with minor adjustments. 20–30% Medium Applications that benefit from managed services, databases needing performance upgrades
Repurchase (Drop & Shop) Replace the existing application with a SaaS or marketplace alternative. Example: Custom CRM → Salesforce, custom monitoring → Datadog. 5–10% Variable Applications with mature SaaS alternatives, undifferentiated capabilities
Refactor (Re-architect) Significantly redesign the application to be cloud-native on AWS. Adopt serverless, microservices, event-driven patterns. 10–20% High Strategic applications requiring scalability, applications with technical debt that justifies investment
Retire Decommission the application entirely. Turn it off, archive data, and redirect users. 10–15% Low Redundant applications, unused services, end-of-life systems
Retain (Revisit Later) Keep the application on GCP for now. Defer migration due to complexity, compliance, or dependency constraints. 5–15% None (deferred) Applications with deep GCP-native dependencies (e.g., Spanner, BigQuery ML), applications under active GCP CUD commitments
Distribution Insight

In a typical enterprise migration, Rehost and Replatform together account for 50–70% of the application portfolio. This means the majority of your migration effort is relatively well-understood and can be executed with established patterns and tooling. The remaining 30–50% requires more nuanced decision-making around Refactor, Repurchase, Retire, and Retain strategies.

2. Discovery: Building the Application Inventory

You cannot classify what you have not inventoried. The discovery phase produces a comprehensive catalog of every application, service, database, and infrastructure component in your GCP environment, along with their interdependencies, resource utilization patterns, and business metadata. Three categories of discovery tools are commonly deployed:

Agent-Based Discovery

AWS Application Discovery Service (ADS) deploys lightweight agents on GCE instances to collect detailed information including CPU/memory/disk utilization, running processes, network connections, and TCP dependency mapping. The agent-based approach provides the highest fidelity data but requires installation on each instance. 

# Install AWS Discovery Agent on a Linux GCE instance
curl -o aws-discovery-agent.tar.gz \
  https://s3-us-west-2.amazonaws.com/aws-discovery-agent.linux/latest/aws-discovery-agent.tar.gz

tar -xzf aws-discovery-agent.tar.gz
sudo bash install -r us-west-2 \
  -k <ACCESS_KEY_ID> \
  -s <SECRET_ACCESS_KEY> \
  -p true \
  -c true

Agentless Discovery

For environments where agent installation is impractical (production databases, compliance-restricted zones), agentless discovery uses API-based collection. AWS Agentless Discovery Connector integrates with vCenter for VMware environments; for GCP-native workloads, use the GCP Asset Inventory API to export a complete resource manifest, then import into AWS Migration Hub for analysis. 

# Export GCP asset inventory for migration planning
gcloud asset export \
  --organization=123456789 \
  --content-type=resource \
  --output-path=gs://migration-staging/gcp-asset-inventory.json \
  --asset-types="compute.googleapis.com/Instance,\
sqladmin.googleapis.com/Instance,\
container.googleapis.com/Cluster,\
storage.googleapis.com/Bucket"

Third-Party Discovery Platforms

Enterprise-grade discovery platforms provide richer analysis, automated dependency mapping, and 6R recommendation engines:

  • CloudAmize: AI-driven discovery with automatic 6R classification, right-sizing recommendations, and cost projections. Supports both GCP and AWS as source/target platforms.
  • Turbonomic (IBM): Application Resource Management platform that maps dependencies in real time and provides migration workload placement recommendations based on performance and cost optimization.
  • Flexera One: Multi-cloud management platform with discovery, cost optimization, and governance capabilities. Particularly strong for hybrid and multi-cloud environments.
  • CAST Highlight: Software intelligence platform that analyzes application source code to assess cloud readiness, technical debt, and refactoring effort.

3. Prioritization Matrices

Once the inventory is complete and each application has a preliminary 6R classification, the next step is prioritization. Not all applications can (or should) migrate simultaneously. A prioritization matrix scores each application across multiple dimensions to determine its position in the migration sequence. The four primary dimensions are:

  1. Business Value: How critical is this application to revenue generation, customer experience, or regulatory compliance? High-value applications need the most careful migration planning but also deliver the greatest ROI when optimized on AWS.
  2. Technical Complexity: How many dependencies does the application have? Does it use GCP-proprietary services (Spanner, BigQuery ML, Apigee) that lack direct AWS equivalents? Complex applications require more planning, testing, and rollback provisions.
  3. Compliance and Data Sensitivity: Does the application process PII, PHI, PCI, or other regulated data? Compliance-sensitive workloads require additional security reviews, data residency verification, and audit trail preservation during migration.
  4. Team Readiness: Does the application team have AWS experience? Are they available to support migration testing and cutover? Applications owned by teams with AWS skills migrate faster and with fewer incidents.

Score each dimension on a 1–5 scale (1 = low, 5 = high). Applications with high business value, low technical complexity, low compliance sensitivity, and high team readiness are ideal early-wave candidates. Applications scoring high on complexity and compliance are deferred to later waves when the team has built confidence and refined runbooks.

4. Wave Planning Methodology

Wave planning organizes the prioritized application inventory into sequenced migration batches, each with defined scope, timeline, success criteria, and rollback procedures. The standard wave progression follows four phases:

Wave 1: Pilot (Weeks 1–4)

The Pilot Wave Is Your Most Important Wave

Select 2–3 low-risk, low-complexity applications for the pilot wave. The goal is not to migrate critical workloads—it is to validate your migration tooling, test your network connectivity between GCP and AWS, verify IAM and security configurations, train the team on operational procedures, and establish baseline metrics for estimating future wave timelines. A successful pilot wave de-risks every subsequent wave. Invest heavily in documentation and retrospectives during this phase.

Pilot wave applications should be stateless or have simple data migration requirements. Good candidates include internal tools, development/staging environments, static websites, and batch-processing jobs. Measure and record: migration duration, downtime (if any), data integrity validation results, and post-migration performance comparison.

Wave 2: Foundation (Weeks 5–12)

The foundation wave migrates 10–15 applications that establish the core infrastructure patterns for the rest of the migration. This includes:

  • Shared services: Active Directory/LDAP, DNS, monitoring, logging, CI/CD pipelines
  • Networking backbone: Transit Gateway, VPN/Direct Connect to on-premises, VPC peering to GCP
  • Security infrastructure: AWS Config rules, GuardDuty, Security Hub, WAF policies
  • First production workloads: Medium-complexity applications that validate the landing zone architecture

Wave 3: Production (Weeks 13–36)

The production wave is the largest phase, migrating the bulk of the application portfolio in sub-waves of 15–25 applications each, executed every 2–3 weeks. Each sub-wave follows a standardized runbook:

  1. Pre-migration validation: Confirm AWS target resources are provisioned, IAM roles are configured, and network routes are verified.
  2. Data replication: Initiate continuous replication using AWS DMS (databases), DataSync (files), or S3 Transfer Acceleration (objects).
  3. Cutover window: Execute DNS switch, validate data integrity, run smoke tests, and monitor for 24–48 hours.
  4. Post-migration optimization: Right-size instances, apply Reserved Instance or Savings Plan coverage, and tune auto-scaling policies.

Wave 4: Optimization (Weeks 37–52)

The final wave focuses on the remaining high-complexity applications (Refactor candidates) and post-migration optimization across the entire portfolio. Activities include:

  • Refactoring monoliths into microservices on EKS or Lambda
  • Migrating analytics workloads from BigQuery to Redshift or Athena
  • Implementing cost governance with AWS Cost Explorer, Budgets, and Trusted Advisor
  • Decommissioning GCP resources and terminating GCP contracts
  • Conducting a comprehensive migration retrospective and documenting lessons learned

5. Migration Readiness Assessment (MRA)

The AWS Migration Readiness Assessment is a structured evaluation framework that scores your organization's preparedness across seven dimensions. Each dimension is scored on a 1–5 maturity scale. Organizations should target a minimum score of 3 in each dimension before beginning production migration waves.

MRA Dimension What It Measures Target Score Common Gaps
Business Case Financial justification, TCO model completeness, executive sponsorship 3+ Missing hidden costs, lack of C-suite buy-in
Landing Zone AWS account structure, networking, security baseline, governance controls 4+ Flat account structure, missing SCPs, no centralized logging
Discovery & Planning Application inventory completeness, dependency mapping, 6R classification 3+ Incomplete inventory, undocumented dependencies, no discovery tooling
Security & Compliance IAM strategy, encryption standards, compliance framework mapping (SOC2, HIPAA, PCI) 4+ GCP-specific compliance controls not mapped to AWS equivalents
Operations Monitoring, alerting, incident response, runbook maturity, on-call processes 3+ GCP-native monitoring not replicated on AWS, missing CloudWatch dashboards
People & Skills AWS certification levels, training plans, staffing adequacy, partner engagement 3+ Team is 100% GCP-certified with no AWS training, no migration partner
Migration Experience Prior migration history, tooling maturity, runbook availability, automation level 3+ First-time migration, manual processes, no IaC templates 
# Example: MRA scoring output (simplified)
mra_scores = {
    "business_case":        4,
    "landing_zone":         3,
    "discovery_planning":   4,
    "security_compliance":  3,
    "operations":           2,   # Gap: monitoring not yet replicated
    "people_skills":        2,   # Gap: team needs AWS training
    "migration_experience": 1    # Gap: first cloud-to-cloud migration
}

avg_score = sum(mra_scores.values()) / len(mra_scores)
print(f"Overall MRA Score: {avg_score:.1f}/5.0")

Do Not Skip the MRA

Organizations that begin production migration waves with MRA scores below 3.0 in critical dimensions (particularly Landing Zone, Security, and Operations) experience 2.4x more migration incidents and 40% longer timelines on average, according to AWS Professional Services benchmarks. Invest the 2–4 weeks needed to close gaps before accelerating.

6. Putting It All Together

Application Portfolio Analysis and Wave Planning are not one-time exercises. The migration backlog is a living artifact that evolves as discovery reveals new dependencies, business priorities shift, and the team's AWS expertise matures. Schedule weekly backlog refinement sessions with application owners, infrastructure leads, and the migration program manager. Update 6R classifications as new information emerges. Re-score prioritization matrices after each wave retrospective. The teams that treat migration as a disciplined, iterative process—rather than a one-shot project—are the teams that deliver on time, on budget, and without service disruptions.

Key Takeaways

  • The 6R framework (Rehost, Replatform, Repurchase, Refactor, Retire, Retain) provides a structured classification for every application in your portfolio. Rehost and Replatform typically account for 50–70% of workloads.
  • Discovery tools—AWS Application Discovery Service, CloudAmize, Turbonomic—automate inventory collection and dependency mapping. Never rely on manual spreadsheets for enterprise-scale migrations.
  • Prioritization matrices scoring business value, technical complexity, compliance sensitivity, and team readiness determine which applications migrate first.
  • Wave planning follows four phases: Pilot (validate tooling), Foundation (establish infrastructure), Production (bulk migration), and Optimization (refactor and tune).
  • The pilot wave is the most important wave. Select 2–3 low-risk applications, document everything, and use the results to calibrate timelines for subsequent waves.
  • The Migration Readiness Assessment (MRA) scores organizational preparedness across 7 dimensions. Target a minimum score of 3 in each dimension before beginning production waves to avoid 2.4x incident rate increases.
